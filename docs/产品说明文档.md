# 产品说明文档

## 一、产品定位

callflow-lab 是复杂外呼任务对话模型自动评测平台。本系统不是训练对话模型，也不是实际外呼业务系统，而是用于评测已有对话模型在复杂任务指令、多轮对话、用户打断和业务约束下的表现。

平台面向模型研发团队、外呼质检团队和业务运营团队，帮助团队在模型上线前或版本迭代后快速发现流程遗漏、约束违规、答非所问、机械重复和业务串场等问题。

## 二、目标用户

- 模型研发人员：需要验证对话模型在复杂指令下的稳定性和回归质量。
- 业务质检人员：需要把人工检查项转为可重复执行的评测用例和规则。
- 产品和运营人员：需要对比不同模型或不同版本在关键任务上的表现。
- 比赛评审和演示人员：需要清晰看到自动评测流程、量化分数和证据链。

## 三、核心能力

### 复杂任务指令导入

系统支持导入复杂外呼任务指令，并解析 Role、Task、Opening Line、Call Flow、Knowledge Points、Constraints 等结构化片段。当前默认脱敏任务包括飞毛腿骑手合同生效外呼评测和课程直播产品升级外呼评测。

### 测试用例管理

每个任务可以关联多个测试用例。用例包含用户画像、初始问题、难度、最大轮数、期望目标、必须满足规则和禁止触发规则，用于覆盖不同外呼对象和风险分支。

### 用户模拟器 Agent

用户模拟器 Agent 负责模拟不同被外呼对象的行为。它会根据任务指令、用户画像和历史对话动态发言，而不是固定模板重复输出。用户可能表现为愿意配合、拒绝、不满、追问、忙碌、开车或技术不熟悉。

### 被测模型接入

平台支持三类 provider：

- `mock_fallback`：本地兜底模型，仅在无 API Key 或演示环境不可用时保证评测流程跑通，非真实 AI 能力。
- `openai_compatible`：真实大模型 API 接入，可接入 Qwen、DeepSeek、GPT、智谱、Moonshot 等兼容 OpenAI 格式的模型。
- `custom_endpoint`：自定义被测模型接口，用于接入外部被测模型服务、企业内部模型或其他团队接口。

历史 `mock_baseline` / `mock_strong` 配置会兼容映射为 `mock_fallback`，但页面不再突出本地 mock 对比概念。

模型配置页展示 provider 名称、类型、启用状态、Base URL、Model Name、Endpoint 和测试连接按钮。API Key 只在后端 `.env` 配置，前端不展示明文。

### 混合评分体系

系统使用“规则评分 + LLM 评估器”的混合评测体系。规则评分检查硬约束，评估器 Agent 结合任务指令和对话记录给出语义级评分。

六项核心指标包括：

- 任务完成度
- 指令遵循率
- 外呼流程覆盖率
- 约束遵守率
- 上下文一致性
- 回复质量

### 可解释报告

报告页展示总分、指标分、评分公式、雷达图、命中规则、失败规则、LLM 评估结论、扣分原因、证据轮次、对话片段、优化建议和完整对话记录。报告围绕被测对话模型在复杂任务指令下的表现，不评价本系统代码质量。

### 批量评测

批量评测支持选择一个或多个任务、测试用例范围、一个或多个模型 provider 和重复次数。默认单模型批量评测即可使用；多 provider 时展示 provider 对比统计，但不强制定义为 A/B 测试。

批量 summary 包含平均分、平均响应时间、通过率、任务得分汇总、指标得分汇总、失败规则 TOP5 和报告列表。

## 四、核心流程

复杂任务指令导入
→ 解析 Role / Task / Opening Line / Flow / Knowledge / Constraints
→ 自动生成测试用例
→ 用户模拟器 Agent 发起多轮对话
→ 被测模型回复
→ 规则评分 + LLM 评估
→ 生成可解释报告
→ 批量统计评测结果

## 五、深度学习和 Agent 使用边界

本系统当前不直接训练大模型，而是通过 LLM API、Prompt 工程和轻量级 Agent 编排实现用户模拟和自动评估。

用户模拟器 Agent 负责模拟不同被外呼对象的行为；评估器 Agent 负责结合任务指令和对话记录进行语义级评分；被测模型主要来自 `openai_compatible` 或 `custom_endpoint` 的真实外部模型接口。

系统真正的 AI 能力来自用户模拟器 Agent 可接入 LLM、LLM-as-a-Judge 评估器、真实大模型 API、自定义外部模型接口、AI 辅助生成测试用例，以及可解释、可量化评测报告。`mock_fallback` 只是本地兜底，不作为系统 AI 能力亮点。

后续如提供人工标注对话数据，可进一步微调用户模拟器或评估器模型，使模拟用户行为更自然、评分结果更贴近人工专家判断。

## 六、典型使用场景

### 模型上线前验证

在新模型版本上线前，团队可以选择默认任务集或自定义任务集进行批量评测，查看平均分、通过率和失败规则 TOP5，判断是否存在阻塞上线的问题。

### 外呼任务回归测试

当业务指令发生变化时，团队可以重新导入任务指令并生成测试用例，对同一模型运行批量评测，确认模型是否仍能覆盖必达流程并遵守新约束。

### 多模型或多版本比较

团队可以选择多个 provider 运行相同任务集，查看不同 provider 的平均分、通过率和延迟表现，为模型选型和版本升级提供依据。

### 评测报告复盘

业务人员可以通过报告页查看具体扣分原因和证据轮次，定位模型在某个对话分支中的问题，并将建议反馈给模型调优或提示词工程环节。

## 七、商业价值

- 降低人工评估成本：自动执行多轮对话、评分和报告生成，减少人工模拟和打分。
- 提高外呼模型回归测试效率：固定任务集可以重复运行，快速发现新版本质量变化。
- 支持复杂任务指令下的自动化验证：不仅看单轮回答，还检查流程、约束和上下文推进。
- 能发现典型模型问题：漏流程、违反约束、回复冗长、答非所问、机械重复、业务串场。
- 支持后续多模型、多版本批量评测：为模型选型、发布门禁和持续质量监控提供基础设施。

## 八、当前边界

- 当前默认评测数据为脱敏任务，真实业务上线前需要接入真实任务指令和人工复核样本。
- 当前批量评测为同步执行，适合演示和中小规模评测；大规模评测可升级为异步任务队列。
- 当前模型配置页主要读取 `.env` 和默认 provider 信息，暂不在前端保存 API Key；`mock_fallback` 只用于演示兜底。
- 当前评估器默认可使用 mock evaluator，接入真实 LLM 评估器后需要复测评分稳定性。
