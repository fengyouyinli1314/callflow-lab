# 测试说明

## 功能测试

功能测试围绕比赛演示主链路进行：

1. 启动后端服务，访问 Swagger。
2. 调用 `GET /api/tasks`，确认存在飞毛腿骑手合同生效外呼和课程直播产品升级外呼任务。
3. 调用 `GET /api/cases?task_id=1`，确认存在对应测试用例且列表没有同签名重复项。
4. 调用 `POST /api/cases`、`PUT /api/cases/{case_id}`、`DELETE /api/cases/{case_id}`，确认人工构建用例链路可用。
5. 调用 `POST /api/cases/generate`，确认返回 AI 生成草稿且不会直接入库。
6. 调用 `POST /api/runs/start`，确认可以生成执行记录和报告。
7. 调用 `GET /api/runs/{run_id}/messages`，确认返回完整多轮对话。
8. 调用 `GET /api/reports/{report_id}`，确认返回可解释报告和 LLM-as-a-Judge 评估结果。

## 接口测试

后端内置 pytest 用例覆盖：

- 任务种子数据是否可见，Excel 不存在时是否有当前外呼任务 fallback 数据。
- 用例是否支持按任务筛选、幂等保存和历史重复清理。
- 手动新增、编辑、删除测试用例是否可用。
- AI 生成用例草稿是否覆盖飞毛腿和课程直播典型用户行为。
- 自动评测是否统一调用 `TargetModelClient.generate_reply(...)`。
- 飞毛腿任务和课程直播任务回复是否避免业务串场。
- 报告是否包含总分、六项指标分、规则评分、LLM 评估、失败规则、证据和建议。
- 批量评测 summary 是否返回平均分、通过率、失败规则 TOP5 和报告列表。
- 模型配置接口是否只暴露 `mock_fallback`、`openai_compatible`、`custom_endpoint`，且不返回 API Key 明文。

运行命令：

```powershell
cd backend
pytest
```

## 评测准确性测试

评测准确性以复杂外呼任务规则覆盖为核心：

- 飞毛腿任务：围绕合同生效、单日/多日配送要求、派单影响、报名排名、退出流程、恶劣天气安全和额外奖励边界进行评测。
- 课程直播任务：围绕负责人确认、发布页升级、标准直播/低延迟直播区别、发布方式、费用、第三方系统路径和企业微信添加边界进行评测。

每条规则会根据当前用例、对话阶段和触发条件进入 active/pending/not applicable 分类，避免把任务级全量规则机械套到每个用例或每一轮。

## 多轮对话流程测试

多轮对话测试重点检查：

- 页面加载后不会自动开始对话，必须选择任务、用例和 provider 后点击“开始评测”。
- 第一轮是否使用测试用例的初始问题。
- 后续轮次是否由用户模拟器 Agent 根据任务指令、用户画像、用例规则和历史对话动态生成。
- 每轮是否保存用户发言、被测模型回复、响应时间、规则评分和用户状态。
- 报告中的对话证据是否与执行记录一致。

## 前端测试

前端验证重点：

- Dashboard、评测任务、测试用例、模型配置、开始评测、完整报告页面不白屏。
- 测试用例页支持新增、编辑、删除、AI 生成草稿和确认保存。
- 开始评测页支持新建用例快捷按钮和编辑当前用例快捷按钮。
- 模型配置页不展示 API Key 明文，并明确 `mock_fallback` 是兜底能力。

构建命令：

```powershell
cd frontend
npm run build
```

## 异常场景测试

建议覆盖以下异常：

- 不存在的 `task_id`。
- 不存在或不属于该任务的 `case_id`。
- 用例表单必填字段为空。
- 重复保存同一 `task_id + name + initial_message` 用例。
- 真实模型 API 未配置或调用失败。
- 前端后端地址配置错误。

系统在模型配置缺失或调用失败时会回退 mock 兜底，保证演示链路可运行。
