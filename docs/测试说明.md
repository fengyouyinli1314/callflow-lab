# 测试说明

## 功能测试

功能测试围绕比赛演示主链路进行：

1. 启动后端服务，访问 Swagger。
2. 调用 `GET /api/tasks`，确认存在 3 个示例任务。
3. 调用 `GET /api/cases?task_id=1`，确认存在对应测试用例。
4. 调用 `POST /api/runs/start`，确认可以生成执行记录和报告。
5. 调用 `GET /api/runs/{run_id}/messages`，确认返回完整多轮对话。
6. 调用 `GET /api/reports/{report_id}`，确认返回可解释报告。

## 接口测试

后端内置 pytest 用例覆盖：

- 任务种子数据是否可见。
- 用例是否支持按任务筛选。
- 是否可以启动自动评测并获取对话记录。
- 报告是否包含 `metric_details`、`failure_cases`、`explainability` 等解释性字段。

运行命令：

```powershell
cd backend
pytest
```

## 评测准确性测试

评测准确性以业务规则覆盖为核心：

- 外卖退款：核实订单号、告知处理时效、安抚用户情绪、避免直接承诺退款。
- 酒店变更：确认订单信息、入住日期、房型库存、差价或费用提示。
- 团购券核销：排查券状态、门店适用范围、有效期、下一步处理建议。

每条规则都会进入报告中的失败规则和失败案例字段，便于复核评分依据。

## 多轮对话流程测试

多轮对话测试重点检查：

- 第一轮是否使用测试用例的初始问题。
- 后续轮次是否由用户模拟器根据用户画像生成。
- 每轮是否保存用户发言、客服回复、响应时间和规则评分。
- 报告中的对话证据是否与执行记录一致。

## 性能测试

当前版本以本地比赛演示为目标，未强制接入压测工具。可通过连续调用 `POST /api/runs/start` 观察响应稳定性和 SQLite 数据写入情况。后续可补充 Locust 脚本，对并发启动评测、报告查询和数据大屏接口进行压力测试。

## 异常场景测试

建议覆盖以下异常：

- 不存在的 `task_id`。
- 不存在或不属于该任务的 `case_id`。
- 空规则列表。
- 真实模型 API 未配置或调用失败。
- 前端后端地址配置错误。

系统在模型配置缺失或调用失败时会回退 mock 模式，保证演示链路可运行。
